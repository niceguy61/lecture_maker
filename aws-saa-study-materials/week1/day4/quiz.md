## 문제 1: EBS 볼륨 타입 선택

**시나리오**: 온라인 게임 서버를 위한 데이터베이스를 구축하려고 합니다. 일관된 높은 IOPS 성능이 필요하고, 지연 시간이 매우 중요합니다.

**질문**: 이 요구사항에 가장 적합한 EBS 볼륨 타입은 무엇입니까?

A) gp3 (General Purpose SSD)  
B) io2 (Provisioned IOPS SSD)  
C) st1 (Throughput Optimized HDD)  
D) sc1 (Cold HDD)

<details>
<summary>정답 및 해설</summary>

**정답: B) io2 (Provisioned IOPS SSD)**

**해설**: 
- io2는 높은 IOPS가 필요한 데이터베이스 워크로드에 최적화되어 있습니다
- 일관된 성능을 제공하며, 최대 64,000 IOPS까지 지원합니다
- 지연 시간이 중요한 애플리케이션에 적합합니다
- gp3는 범용이지만 일관된 고성능이 필요한 경우에는 io2가 더 적합합니다
- st1과 sc1은 HDD 기반으로 IOPS 성능이 낮습니다

**관련 개념**: EBS 볼륨 타입, IOPS, 데이터베이스 성능 요구사항
</details>

## 문제 2: EBS와 인스턴스 스토어 비교

**질문**: 다음 중 인스턴스 스토어(Instance Store)의 특징으로 **올바르지 않은** 것은?

A) EC2 인스턴스를 중지하면 데이터가 삭제됩니다  
B) EBS보다 높은 I/O 성능을 제공할 수 있습니다  
C) 인스턴스와 함께 자동으로 백업됩니다  
D) 임시 데이터나 캐시 저장에 적합합니다

<details>
<summary>정답 및 해설</summary>

**정답: C) 인스턴스와 함께 자동으로 백업됩니다**

**해설**: 
- 인스턴스 스토어는 임시 스토리지로, 백업 기능을 제공하지 않습니다
- 인스턴스를 중지하거나 종료하면 모든 데이터가 영구적으로 삭제됩니다
- 물리적으로 인스턴스에 직접 연결되어 있어 매우 높은 I/O 성능을 제공합니다
- 캐시, 임시 파일, 버퍼 등의 용도로 사용하기 적합합니다
- 영구적인 데이터 저장이 필요한 경우에는 EBS를 사용해야 합니다

**관련 개념**: 인스턴스 스토어 vs EBS, 임시 스토리지, 데이터 영속성
</details>

## 문제 3: EBS 스냅샷 이해

**시나리오**: 중요한 데이터가 저장된 EBS 볼륨의 백업을 생성하려고 합니다. 또한 이 백업을 다른 리전에서도 사용할 수 있어야 합니다.

**질문**: 이 요구사항을 만족하는 가장 적절한 방법은 무엇입니까?

A) EBS 볼륨을 다른 인스턴스에 연결하여 복사  
B) EBS 스냅샷을 생성하고 다른 리전으로 복사  
C) 인스턴스 전체를 AMI로 생성  
D) EFS를 사용하여 파일을 복사

<details>
<summary>정답 및 해설</summary>

**정답: B) EBS 스냅샷을 생성하고 다른 리전으로 복사**

**해설**: 
- EBS 스냅샷은 특정 시점의 볼륨 백업을 S3에 저장합니다
- 스냅샷은 다른 리전으로 복사할 수 있어 재해 복구에 유용합니다
- 스냅샷에서 새로운 EBS 볼륨을 생성할 수 있습니다
- 증분 백업 방식으로 비용 효율적입니다
- A는 복잡하고 비효율적이며, C는 전체 인스턴스 백업, D는 파일 시스템 서비스로 목적이 다릅니다

**관련 개념**: EBS 스냅샷, 백업 전략, 리전 간 복사, 재해 복구
</details>

## 문제 4: EC2 인스턴스 타입 선택

**시나리오**: 웹 애플리케이션을 운영하는데, 평소에는 CPU 사용률이 낮지만 특정 시간대에 트래픽이 급증하여 높은 CPU 성능이 필요합니다. 비용 효율성도 중요합니다.

**질문**: 이 시나리오에 가장 적합한 EC2 인스턴스 타입은 무엇입니까?

A) C5 (Compute Optimized)  
B) M5 (General Purpose)  
C) T3 (Burstable Performance)  
D) R5 (Memory Optimized)

<details>
<summary>정답 및 해설</summary>

**정답: C) T3 (Burstable Performance)**

**해설**: 
- T3는 버스터블 성능 인스턴스로, 평소에는 기본 성능으로 운영되다가 필요시 CPU 크레딧을 사용하여 성능을 폭발적으로 높일 수 있습니다
- 평소 CPU 사용률이 낮은 워크로드에 비용 효율적입니다
- 트래픽 패턴이 예측 가능하고 간헐적인 고성능이 필요한 경우에 적합합니다
- C5는 지속적인 고성능이 필요한 경우, M5는 균형잡힌 워크로드, R5는 메모리 집약적 워크로드에 적합합니다

**관련 개념**: 버스터블 성능, CPU 크레딧, 비용 최적화, 워크로드 패턴
</details>

## 문제 5: EFS vs EBS 비교

**시나리오**: 여러 EC2 인스턴스에서 동시에 접근해야 하는 공유 파일 시스템이 필요합니다. 파일 시스템은 자동으로 확장되어야 하고, 여러 가용 영역에서 접근 가능해야 합니다.

**질문**: 이 요구사항에 가장 적합한 AWS 스토리지 서비스는 무엇입니까?

A) EBS (Elastic Block Store)  
B) EFS (Elastic File System)  
C) S3 (Simple Storage Service)  
D) Instance Store

<details>
<summary>정답 및 해설</summary>

**정답: B) EFS (Elastic File System)**

**해설**: 
- EFS는 여러 EC2 인스턴스가 동시에 접근할 수 있는 완전 관리형 NFS 파일 시스템입니다
- 자동으로 확장되며, 사용한 만큼만 비용을 지불합니다
- 여러 가용 영역에 걸쳐 데이터를 복제하여 고가용성을 제공합니다
- POSIX 호환 파일 시스템으로 기존 애플리케이션과 쉽게 통합됩니다
- EBS는 한 번에 하나의 인스턴스만 연결 가능, S3는 객체 스토리지, Instance Store는 임시 스토리지입니다

**관련 개념**: 공유 파일 시스템, NFS, 다중 인스턴스 접근, 자동 확장
</details>
