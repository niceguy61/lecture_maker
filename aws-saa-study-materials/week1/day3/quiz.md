## 문제 1: EC2 인스턴스 유형 (객관식)

**문제**: 다음 중 CPU 집약적인 워크로드에 가장 적합한 EC2 인스턴스 패밀리는 무엇입니까?

A) T3 (버스터블 성능)  
B) M5 (범용)  
C) C5 (컴퓨팅 최적화)  
D) R5 (메모리 최적화)

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: C) C5 (컴퓨팅 최적화)**

**해설:**
- **C5 인스턴스**는 컴퓨팅 최적화 패밀리로, 고성능 프로세서와 높은 CPU 대 메모리 비율을 제공합니다.
- CPU 집약적인 워크로드(고성능 웹 서버, 과학 계산, 게임 서버 등)에 최적화되어 있습니다.

**다른 선택지 설명:**
- **T3**: 버스터블 성능으로 가변적 워크로드에 적합하지만 지속적인 고성능 CPU 작업에는 부적합
- **M5**: 범용으로 균형잡힌 성능을 제공하지만 CPU 특화는 아님
- **R5**: 메모리 최적화로 메모리 집약적 작업에 특화

**관련 개념**: 인스턴스 패밀리별 특징, 워크로드 최적화
</details>

## 문제 2: 보안 그룹 (시나리오 기반)

**문제**: 웹 서버를 운영하는 EC2 인스턴스에 대한 보안 그룹을 설정하고 있습니다. 다음 중 **가장 안전한** 인바운드 규칙 설정은 무엇입니까?

A) 모든 포트(0-65535)를 0.0.0.0/0에서 허용  
B) SSH(22), HTTP(80), HTTPS(443)를 0.0.0.0/0에서 허용  
C) SSH(22)는 내 IP에서만, HTTP(80), HTTPS(443)는 0.0.0.0/0에서 허용  
D) SSH(22), HTTP(80)만 내 IP에서 허용

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: C) SSH(22)는 내 IP에서만, HTTP(80), HTTPS(443)는 0.0.0.0/0에서 허용**

**해설:**
보안 그룹 설정 시 **최소 권한 원칙**을 적용해야 합니다:

- **SSH (포트 22)**: 관리 목적이므로 특정 IP(내 IP)에서만 접근 허용
- **HTTP/HTTPS (포트 80/443)**: 웹 서비스 제공이 목적이므로 모든 곳에서 접근 허용

**다른 선택지 문제점:**
- **A**: 모든 포트 개방은 심각한 보안 위험
- **B**: SSH를 모든 곳에서 허용하면 브루트 포스 공격 위험
- **D**: 웹 서비스가 외부에서 접근 불가능

**보안 모범 사례:**
- 관리 포트(SSH, RDP)는 특정 IP로 제한
- 서비스 포트는 필요한 범위에서만 허용
- 불필요한 포트는 절대 개방하지 않음

**관련 개념**: 보안 그룹, 최소 권한 원칙, 네트워크 보안
</details>

## 문제 3: EC2 인스턴스 상태 (객관식)

**문제**: EC2 인스턴스를 "중지(Stop)"했을 때 발생하는 상황으로 **올바른** 것은?

A) 인스턴스가 완전히 삭제되고 모든 데이터가 손실됩니다  
B) 퍼블릭 IP 주소가 유지되고 컴퓨팅 요금이 계속 부과됩니다  
C) EBS 루트 볼륨은 유지되지만 퍼블릭 IP 주소는 해제됩니다  
D) 인스턴스 스토어의 데이터는 유지되지만 EBS 볼륨은 삭제됩니다

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: C) EBS 루트 볼륨은 유지되지만 퍼블릭 IP 주소는 해제됩니다**

**해설:**
EC2 인스턴스 **중지(Stop)** 시:

**유지되는 것:**
- EBS 루트 볼륨 및 추가 EBS 볼륨
- 인스턴스 설정 및 메타데이터
- 보안 그룹, 키 페어 등 연결된 리소스

**변경되는 것:**
- 퍼블릭 IP 주소 해제 (재시작 시 새 IP 할당)
- 컴퓨팅 요금 부과 중단 (스토리지 요금은 계속)
- 인스턴스 스토어 데이터 손실 (있는 경우)

**다른 선택지 설명:**
- **A**: 이는 "종료(Terminate)"의 특징
- **B**: 중지 시 퍼블릭 IP는 해제되고 컴퓨팅 요금 부과 중단
- **D**: 반대임 (인스턴스 스토어 손실, EBS 유지)

**관련 개념**: 인스턴스 생명주기, 중지 vs 종료, EBS vs 인스턴스 스토어
</details>

## 문제 4: EC2 요금 모델 (시나리오 기반)

**문제**: 스타트업에서 웹 애플리케이션을 운영하고 있습니다. 트래픽이 예측하기 어렵고, 때로는 몇 시간 동안 사용하지 않을 수도 있습니다. 또한 애플리케이션이 갑자기 중단되어도 큰 문제가 없습니다. 이 상황에서 **가장 비용 효율적인** EC2 요금 모델은?

A) 온디맨드 인스턴스  
B) 예약 인스턴스 (1년 약정)  
C) 스팟 인스턴스  
D) 전용 호스트

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: C) 스팟 인스턴스**

**해설:**
주어진 조건 분석:
- ✅ **예측 불가능한 트래픽**: 스팟 인스턴스의 가변적 특성과 부합
- ✅ **간헐적 사용**: 필요할 때만 사용하는 패턴
- ✅ **중단 허용**: 스팟 인스턴스의 가장 중요한 조건

**스팟 인스턴스 장점:**
- 온디맨드 대비 최대 90% 할인
- 유연한 사용 패턴에 적합
- 내결함성 애플리케이션에 이상적

**다른 선택지 분석:**
- **A) 온디맨드**: 유연하지만 비용이 높음
- **B) 예약 인스턴스**: 예측 불가능한 워크로드에 부적합
- **D) 전용 호스트**: 가장 비싸고 과도한 사양

**주의사항:**
- 스팟 인스턴스는 2분 전 통지 후 중단 가능
- 중요한 프로덕션 워크로드에는 부적합
- 배치 작업, 개발/테스트 환경에 적합

**관련 개념**: EC2 요금 모델, 스팟 인스턴스, 비용 최적화
</details>

## 문제 5: 실습 기반 문제 (종합)

**문제**: 실습에서 EC2 인스턴스에 Apache 웹 서버를 설치한 후, 웹 브라우저에서 접속이 되지 않습니다. SSH 접속은 정상적으로 작동하고, 인스턴스 내부에서 `curl localhost`는 성공합니다. 가장 가능성이 높은 원인과 해결방법은?

A) Apache 서비스가 실행되지 않음 → `sudo systemctl start httpd`  
B) 보안 그룹에서 HTTP(80) 포트가 허용되지 않음 → 인바운드 규칙에 HTTP 추가  
C) 키 페어 설정 문제 → 새로운 키 페어 생성  
D) 인스턴스 유형이 웹 서버에 적합하지 않음 → 더 큰 인스턴스로 변경

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: B) 보안 그룹에서 HTTP(80) 포트가 허용되지 않음 → 인바운드 규칙에 HTTP 추가**

**해설:**
문제 상황 분석:
- ✅ SSH 접속 성공 → 인스턴스와 네트워크 연결은 정상
- ✅ `curl localhost` 성공 → Apache 서비스는 정상 실행 중
- ❌ 외부에서 웹 접속 실패 → 방화벽(보안 그룹) 문제 가능성 높음

**문제 해결 과정:**
1. **보안 그룹 확인**: EC2 Console → 인스턴스 → 보안 탭
2. **인바운드 규칙 확인**: HTTP (포트 80) 규칙 존재 여부
3. **규칙 추가**: 유형=HTTP, 포트=80, 소스=0.0.0.0/0

**다른 선택지 분석:**
- **A**: `curl localhost` 성공으로 Apache 정상 실행 확인됨
- **C**: SSH 접속 성공으로 키 페어 문제 아님
- **D**: t3.micro도 웹 서버 운영에 충분함

**보안 그룹 특징:**
- 상태 저장 방화벽 (Stateful)
- 기본적으로 모든 인바운드 트래픽 차단
- 허용 규칙만 설정 가능 (거부 규칙 없음)
- 변경 사항 즉시 적용

**실무 팁:**
- 웹 서비스 문제 시 보안 그룹을 가장 먼저 확인
- 로컬 테스트(`curl localhost`)로 서비스 상태 확인
- 단계별 문제 해결 접근법 사용

**관련 개념**: 보안 그룹, 네트워크 문제 해결, Apache 웹 서버
</details>
