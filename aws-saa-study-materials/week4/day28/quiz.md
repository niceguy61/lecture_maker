## 문제 2: IAM 정책 평가 및 권한 관리 (복수 정답)

다음 중 IAM 정책 평가에서 **올바른** 설명은? (2개 선택)

A) 명시적 허용이 명시적 거부보다 우선순위가 높다
B) 명시적 거부가 모든 허용보다 우선순위가 높다
C) 기본적으로 모든 작업이 거부된다
D) 여러 정책이 충돌할 때 가장 관대한 정책이 적용된다
E) 리소스 기반 정책이 항상 자격 증명 기반 정책보다 우선한다

<details>
<summary>정답 및 해설 보기</summary>

**정답: B, C**

**해설:**
IAM 정책 평가는 보안을 최우선으로 하는 명확한 규칙을 따릅니다.

**정책 평가 순서:**
1. **기본 거부 (Implicit Deny)**: 모든 작업이 기본적으로 거부됨
2. **명시적 허용 (Explicit Allow)**: 정책에서 명시적으로 허용된 작업
3. **명시적 거부 (Explicit Deny)**: 최우선 - 모든 허용을 무효화

**선택지 분석:**
- **A) 틀림**: 명시적 거부가 항상 최우선
- **B) 정답**: 명시적 거부는 절대적 우선순위
- **C) 정답**: 기본적으로 모든 작업 거부 (Least Privilege 원칙)
- **D) 틀림**: 가장 제한적인 정책이 적용됨
- **E) 틀림**: 정책 유형보다는 명시적 거부 여부가 중요

**실무 예시:**
```json
{
  "Effect": "Deny",
  "Action": "s3:DeleteObject",
  "Resource": "*"
}
```
위 정책이 있으면 다른 모든 허용 정책과 관계없이 S3 객체 삭제가 차단됩니다.

**관련 개념**: IAM 정책 평가, 최소 권한 원칙, 보안 모범 사례
</details>

- 30-90일: 가끔 액세스
- 90일-7년: 거의 액세스 안함, 빠른 복구 필요

C) Standard → IA → Glacier Instant Retrieval → Deep Archive

<summary>정답 및 해설 보기</summary>

## 문제 5: RDS 고가용성 및 성능 최적화 (복수 정답)

다음 중 RDS 데이터베이스의 **고가용성과 성능을 모두 향상**시키는 방법은? (2개 선택)

A) Multi-AZ 배포 설정
B) Read Replica 생성
C) 인스턴스 타입을 더 큰 크기로 변경
D) 자동 백업 비활성화
E) Connection Pooling 구현

<details>
<summary>정답 및 해설 보기</summary>

**정답: A, B**

**해설:**
고가용성과 성능을 동시에 향상시키는 RDS 기능들을 이해해야 합니다.

**선택지 분석:**

**A) Multi-AZ 배포** ✅:
- **고가용성**: 자동 장애 조치, 99.95% SLA
- **성능**: 백업이 Standby에서 수행되어 Primary 성능 영향 최소화
- 동기식 복제로 데이터 일관성 보장

**B) Read Replica** ✅:
- **고가용성**: 읽기 트래픽 분산으로 Primary 부하 감소
- **성능**: 읽기 성능 대폭 향상 (최대 15개 복제본)
- 비동기식 복제로 Primary 성능 영향 최소화

**C) 인스턴스 타입 변경**:
- 성능 향상은 되지만 고가용성과는 무관
- 비용 증가 요인

**D) 자동 백업 비활성화**:
- ❌ 고가용성 저하 (복구 불가능)
- 성능 향상 효과 미미

**E) Connection Pooling**:
- 성능 향상은 되지만 고가용성과는 무관
- 애플리케이션 레벨 최적화

**아키텍처 예시:**
```
Primary DB (Multi-AZ)
├── Standby DB (다른 AZ) - 고가용성
└── Read Replica 1, 2, 3 - 성능 향상
```

**관련 개념**: RDS Multi-AZ, Read Replica, 고가용성 설계
</details>

## 문제 6: VPC 네트워킹 및 보안

**시나리오:**
- 웹 서버: 인터넷에서 접근 가능해야 함
- 애플리케이션 서버: 인터넷 접근 불가, 아웃바운드 인터넷 필요
- 데이터베이스: 완전히 격리

A) 모든 서버를 퍼블릭 서브넷에 배치
B) 웹: 퍼블릭, 앱: 프라이빗 + NAT Gateway, DB: 프라이빗
C) 웹: 퍼블릭, 앱: 프라이빗 + NAT Instance, DB: 퍼블릭
D) 모든 서버를 프라이빗 서브넷에 배치 + VPN
E) 웹: 퍼블릭, 앱/DB: 프라이빗 + Internet Gateway

<details>
<summary>정답 및 해설 보기</summary>

**정답: B**

**해설:**
3-Tier 아키텍처의 보안 모범 사례를 적용한 네트워크 설계입니다.

**최적 구성:**

**웹 서버 - 퍼블릭 서브넷:**
- Internet Gateway를 통한 인바운드/아웃바운드 인터넷 액세스
- 보안 그룹으로 HTTP/HTTPS 포트만 허용
- 퍼블릭 IP 또는 Elastic IP 할당

**애플리케이션 서버 - 프라이빗 서브넷 + NAT Gateway:**
- 인바운드 인터넷 액세스 차단 (보안 강화)
- NAT Gateway를 통한 아웃바운드 인터넷 액세스 (패치, 업데이트)
- 웹 서버에서만 접근 가능하도록 보안 그룹 설정

**데이터베이스 - 프라이빗 서브넷:**
- 완전한 인터넷 격리
- 애플리케이션 서버에서만 접근 가능
- DB 서브넷 그룹으로 추가 보안

**보안 계층:**
```
Internet → IGW → ALB (퍼블릭) → 웹서버 (퍼블릭)
                                    ↓
                              앱서버 (프라이빗) → NAT GW → IGW
                                    ↓
                              DB (프라이빗, 완전 격리)
```

**다른 선택지 문제점:**
- **A)**: DB가 인터넷에 노출되어 보안 위험
- **C)**: DB가 퍼블릭 서브넷에 있어 위험
- **D)**: 웹 서버가 인터넷 접근 불가
- **E)**: Internet Gateway는 퍼블릭 서브넷에만 적용

**관련 개념**: VPC 보안 설계, 3-Tier 아키텍처, NAT Gateway
</details>

## 문제 7: CloudFront 및 성능 최적화

다음 중 CloudFront를 사용한 **성능 최적화** 전략으로 **가장 효과적인** 조합은?

A) 모든 콘텐츠를 동일한 TTL로 설정
B) Origin Request Policy + Cache Policy + 압축 활성화
C) 단일 Origin만 사용하여 단순화
D) 모든 쿼리 스트링을 캐시 키에 포함
E) Edge Location을 특정 지역으로 제한

<details>
<summary>정답 및 해설 보기</summary>

**정답: B**

**해설:**
CloudFront의 고급 기능을 활용한 종합적인 성능 최적화 전략입니다.

**최적화 구성 요소:**

**Origin Request Policy:**
- Origin으로 전달할 헤더, 쿼리, 쿠키 제어
- 불필요한 데이터 전송 최소화
- 캐시 적중률 향상

**Cache Policy:**
- 캐시 키 구성 최적화
- TTL 설정으로 콘텐츠별 캐싱 전략
- 압축 설정으로 전송 데이터 크기 감소

**압축 활성화:**
- Gzip/Brotli 압축으로 대역폭 절약
- 텍스트 기반 콘텐츠 70-90% 크기 감소
- 전송 시간 단축

**성능 향상 효과:**
```
압축: 70-90% 크기 감소
캐시 정책: 90%+ 캐시 적중률
Origin 정책: 불필요한 요청 50% 감소
```

**다른 선택지 문제점:**
- **A)**: 콘텐츠별 특성 무시로 비효율적
- **C)**: 다중 Origin 활용으로 지연시간 최적화 가능
- **D)**: 캐시 적중률 크게 저하
- **E)**: 글로벌 배포의 이점 상실

**실무 설정 예시:**
- 정적 파일: TTL 1년, 압축 활성화
- API 응답: TTL 5분, 쿼리 스트링 기반 캐싱
- 동적 콘텐츠: TTL 0, Origin Request Policy로 최적화

**관련 개념**: CloudFront 최적화, 캐시 전략, 성능 튜닝
</details>


**서버리스 장점:**
- **확장성**: 자동 확장, 무제한 처리량
- **비용 효율성**: 사용량 기반 과금, 유휴 비용 없음
- **운영 부담 없음**: 서버 관리 불필요
- **고가용성**: AWS 관리형 서비스의 내장 가용성

**다른 선택지 분석:**
- **A, C, D**: EC2/ECS 사용으로 서버리스 아니며 운영 부담 존재
- **E**: 정적 콘텐츠 배포용으로 이미지 처리 기능 없음

**관련 개념**: 서버리스 아키텍처, 이벤트 기반 처리, AWS Lambda
</details>

## 문제 15: 종합 아키텍처 시나리오

다음 복합 요구사항을 **모두 만족**하는 아키텍처는?

**요구사항:**
- 글로벌 사용자 대상 (미국, 유럽, 아시아)
- 마이크로서비스 아키텍처
- 컨테이너 기반 배포
- 데이터 지역별 격리 (GDPR 준수)
- 99.99% 가용성
- 자동 확장 및 배포

A) 단일 리전 + EKS + RDS + CloudFront
B) 다중 리전 + ECS Fargate + Aurora Global + Route 53 + CloudFront
C) 다중 리전 + EKS + 리전별 RDS + Route 53 Geolocation + CloudFront
D) Lambda + API Gateway + DynamoDB Global Tables
E) EC2 Auto Scaling + ALB + RDS Multi-AZ

<details>
<summary>정답 및 해설 보기</summary>

**정답: C**

**해설:**
복합적인 글로벌 엔터프라이즈 요구사항을 모두 만족하는 완전한 아키텍처입니다.

**아키텍처 구성 요소:**

**다중 리전 배포:**
- 미국(us-east-1), 유럽(eu-west-1), 아시아(ap-northeast-1)
- 각 리전에 완전한 서비스 스택 배포
- 리전 장애 시 다른 리전으로 자동 전환

**EKS (Kubernetes):**
- 마이크로서비스 아키텍처에 최적화
- 컨테이너 오케스트레이션 및 자동 확장
- 서비스 메시, 로드 밸런싱, 서비스 디스커버리

**리전별 RDS:**
- 데이터 지역별 격리로 GDPR 준수
- 각 리전의 사용자 데이터를 해당 리전에만 저장
- Multi-AZ로 리전 내 고가용성 보장

**Route 53 Geolocation:**
- 사용자 위치 기반 자동 라우팅
- 지연시간 최소화 및 규정 준수
- Health Check로 장애 시 자동 전환

**CloudFront:**
- 글로벌 CDN으로 정적 콘텐츠 배포
- 엣지 로케이션에서 캐싱으로 성능 향상
- DDoS 보호 및 보안 강화

**요구사항 충족도:**
```
✅ 글로벌 서비스: 다중 리전 + CloudFront
✅ 마이크로서비스: EKS + 컨테이너
✅ 컨테이너 배포: EKS 네이티브 지원
✅ 데이터 격리: 리전별 RDS
✅ 99.99% 가용성: Multi-AZ + 다중 리전
✅ 자동 확장: EKS HPA + Cluster Autoscaler
✅ 자동 배포: EKS + CI/CD 파이프라인
```

**아키텍처 다이어그램:**
```
CloudFront (글로벌 CDN)
    ↓
Route 53 (Geolocation Routing)
    ↓
┌─────────────┬─────────────┬─────────────┐
│   US 리전    │   EU 리전    │  Asia 리전   │
│ EKS Cluster │ EKS Cluster │ EKS Cluster │
│ RDS (US)    │ RDS (EU)    │ RDS (Asia)  │
└─────────────┴─────────────┴─────────────┘
```

**다른 선택지 분석:**
- **A)**: 단일 리전으로 글로벌 요구사항 미충족
- **B)**: Aurora Global은 데이터 격리 요구사항 위반
- **D)**: 서버리스이지만 마이크로서비스 복잡성 처리 제한적
- **E)**: 컨테이너 기반이 아니며 글로벌 배포 고려 없음

**관련 개념**: 글로벌 아키텍처, 마이크로서비스, 데이터 주권, GDPR 준수
</details>
