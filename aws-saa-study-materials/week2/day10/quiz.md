## 문제 1: RDS 기본 개념

**Amazon RDS의 주요 장점으로 올바르지 않은 것은?**

A) 자동 백업 및 복원 기능 제공  
B) 운영 체제 및 데이터베이스 소프트웨어 자동 패치  
C) 데이터베이스 스키마 자동 최적화  
D) Multi-AZ 배포를 통한 고가용성 제공  

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: C) 데이터베이스 스키마 자동 최적화**

**해설:**
Amazon RDS는 완전 관리형 데이터베이스 서비스로 다음과 같은 장점을 제공합니다:
- **A) 자동 백업**: Point-in-time 복구를 위한 자동 백업 및 스냅샷 기능
- **B) 자동 패치**: OS 및 데이터베이스 엔진의 보안 패치 자동 적용
- **D) Multi-AZ**: 고가용성을 위한 다중 가용 영역 배포

하지만 **데이터베이스 스키마 최적화**는 애플리케이션 로직과 비즈니스 요구사항에 따라 달라지는 부분으로, 개발자가 직접 설계하고 관리해야 합니다. RDS는 인프라 관리를 자동화하지만, 데이터베이스 설계나 쿼리 최적화는 사용자의 책임입니다.

**관련 개념:**
- RDS는 "관리형 서비스"로 인프라 계층을 관리
- 애플리케이션 계층(스키마, 쿼리)은 사용자 관리 영역
</details>

## 문제 2: RDS 엔진 선택

**다음 시나리오에서 가장 적합한 RDS 엔진은?**

**시나리오**: 스타트업에서 웹 애플리케이션을 개발 중입니다. JSON 데이터를 자주 다루고, 복잡한 쿼리와 분석 작업이 필요하며, 오픈소스 솔루션을 선호합니다. 또한 향후 확장성을 고려해야 합니다.

A) MySQL  
B) PostgreSQL  
C) Oracle  
D) SQL Server  

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: B) PostgreSQL**

**해설:**
시나리오의 요구사항을 분석하면:

1. **JSON 데이터 처리**: PostgreSQL은 네이티브 JSON 및 JSONB 데이터 타입을 지원하여 JSON 데이터를 효율적으로 저장하고 쿼리할 수 있습니다.

2. **복잡한 쿼리와 분석**: PostgreSQL은 고급 SQL 기능, 윈도우 함수, CTE(Common Table Expressions), 배열 처리 등을 지원합니다.

3. **오픈소스**: PostgreSQL은 완전한 오픈소스 데이터베이스입니다.

4. **확장성**: PostgreSQL은 파티셔닝, 인덱스 최적화, 확장 모듈 등을 통해 뛰어난 확장성을 제공합니다.

**다른 옵션들:**
- **A) MySQL**: JSON 지원이 제한적이고 복잡한 분석 쿼리에는 상대적으로 부족
- **C) Oracle**: 상용 라이선스 비용이 높고 스타트업에 부담
- **D) SQL Server**: Microsoft 생태계에 특화되어 있고 라이선스 비용 발생

**실무 팁**: PostgreSQL은 "세계에서 가장 고급 오픈소스 데이터베이스"로 불리며, 특히 복잡한 데이터 처리와 분석에 강점을 가집니다.
</details>

## 문제 3: Multi-AZ vs Read Replica

**Multi-AZ 배포와 Read Replica의 차이점에 대한 설명으로 올바른 것은?**

A) Multi-AZ는 읽기 성능 향상을 위해, Read Replica는 고가용성을 위해 사용된다  
B) Multi-AZ는 동기 복제를, Read Replica는 비동기 복제를 사용한다  
C) Multi-AZ의 Standby 인스턴스는 읽기 전용으로 사용할 수 있다  
D) Read Replica는 같은 리전 내에서만 생성 가능하다  

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: B) Multi-AZ는 동기 복제를, Read Replica는 비동기 복제를 사용한다**

**해설:**

**Multi-AZ 배포:**
- **목적**: 고가용성 및 재해 복구
- **복제 방식**: 동기 복제 (데이터 손실 없음)
- **장애조치**: 자동 (1-2분 내)
- **Standby 사용**: 읽기 전용으로 사용 불가 (대기 상태만)
- **위치**: 같은 리전 내 다른 AZ

**Read Replica:**
- **목적**: 읽기 성능 향상 및 읽기 워크로드 분산
- **복제 방식**: 비동기 복제 (약간의 지연 발생)
- **사용**: 읽기 전용 쿼리 처리 가능
- **위치**: 같은 리전 또는 다른 리전 (Cross-Region)

**틀린 옵션들:**
- **A)**: 목적이 반대로 설명됨
- **C)**: Multi-AZ의 Standby는 읽기용으로 사용 불가
- **D)**: Read Replica는 Cross-Region 생성 가능

**실무 활용:**
- 고가용성이 필요하면 Multi-AZ
- 읽기 성능 향상이 필요하면 Read Replica
- 둘 다 함께 사용 가능
</details>

## 문제 4: RDS 스토리지 유형

**다음 중 I/O 집약적인 OLTP(Online Transaction Processing) 워크로드에 가장 적합한 RDS 스토리지 유형은?**

A) 마그네틱 스토리지 (Magnetic)  
B) 범용 SSD (gp2)  
C) 프로비저닝된 IOPS SSD (io1)  
D) 콜드 HDD (sc1)  

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: C) 프로비저닝된 IOPS SSD (io1)**

**해설:**

**OLTP 워크로드의 특징:**
- 높은 트랜잭션 처리량 요구
- 일관된 낮은 지연시간 필요
- 예측 가능한 I/O 성능 요구
- 동시 다중 사용자 접근

**스토리지 유형별 특성:**

**A) 마그네틱 스토리지:**
- 이전 세대 기술
- 낮은 IOPS (100 IOPS)
- 높은 지연시간
- OLTP에 부적합

**B) 범용 SSD (gp2):**
- 기본 3 IOPS/GB
- 버스트 성능 (최대 3,000 IOPS)
- 일반적인 워크로드에 적합
- I/O 집약적 OLTP에는 제한적

**C) 프로비저닝된 IOPS SSD (io1/io2):**
- 일관된 고성능 I/O 제공
- 최대 80,000 IOPS (io2)
- 예측 가능한 성능
- **I/O 집약적 OLTP에 최적**

**D) 콜드 HDD (sc1):**
- RDS에서 지원하지 않음 (EBS 전용)

**성능 비교:**
| 스토리지 유형 | 최대 IOPS | 지연시간 | 용도 |
|---------------|-----------|----------|------|
| Magnetic | 100 | 높음 | 레거시 |
| gp2 | 16,000 | 중간 | 일반 |
| io1/io2 | 80,000 | 낮음 | 고성능 |

**실무 권장사항:**
- 일반적인 웹 애플리케이션: gp2
- 고성능 OLTP: io1/io2
- 개발/테스트: gp2 또는 magnetic
</details>

## 문제 5: RDS 백업 및 복원

**Amazon RDS의 백업 및 복원에 대한 설명으로 올바르지 않은 것은?**

A) 자동 백업은 최대 35일까지 보존할 수 있다  
B) Point-in-time 복구는 5분 단위로 가능하다  
C) 수동 스냅샷은 DB 인스턴스 삭제 후에도 유지된다  
D) Multi-AZ 환경에서는 Primary 인스턴스에서 백업이 수행된다  

<details>
<summary><strong>정답 및 해설</strong></summary>

**정답: D) Multi-AZ 환경에서는 Primary 인스턴스에서 백업이 수행된다**

**해설:**

**RDS 백업 메커니즘:**

**자동 백업:**
- **A) 보존 기간**: 0-35일 설정 가능 (기본 7일)
- **B) Point-in-time 복구**: 5분 단위로 정확한 시점 복구 가능
- **백업 수행 위치**: Multi-AZ 환경에서는 **Standby 인스턴스**에서 수행

**수동 스냅샷:**
- **C) 영구 보존**: DB 인스턴스 삭제 후에도 명시적으로 삭제할 때까지 유지
- 다른 리전으로 복사 가능
- 무제한 보존 가능

**Multi-AZ 백업의 장점:**
- **Primary 인스턴스 성능 영향 없음**
- Standby에서 백업 수행으로 I/O 부하 분산
- 백업 중에도 Primary는 정상 서비스 제공

**백업 전략 권장사항:**
```
자동 백업: 일일 운영 복구용 (7-35일)
수동 스냅샷: 중요 변경 전, 장기 보관용
Cross-Region 복사: 재해 복구용
```

**실무 팁:**
- 백업 윈도우는 낮은 활동 시간대로 설정
- Multi-AZ 사용시 백업으로 인한 성능 저하 없음
- 중요한 변경 작업 전에는 반드시 수동 스냅샷 생성
</details>
